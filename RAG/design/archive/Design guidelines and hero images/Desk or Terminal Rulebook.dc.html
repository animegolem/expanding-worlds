Desk or Terminal — the rulebook

When to use skeuomorphism, when to use flat mono, with every example in the app cataloged. This is a repo-onboarding document: read it before your first UI pull request. If your change contradicts a rule here, the change is wrong or this document needs an amendment — those are the only two options; silent exceptions are how design systems die.

Compiled 2026-07-07 at RFC rev 0.54 · child of STYLE-GUIDE.md §1/§6 · examples cite their ratifying document

1 · The one sentence

Chrome is a terminal; the world is a desk. The canvas holds objects with gentle physics — they lift, settle, tear, press, burn. The UI around them is a flat, mono-flavored terminal that names itself and never emotes. Every pixel you ship belongs to exactly one of these two materials, and mixing them in one element is the cardinal sin.

2 · The test — three questions

1. Does the user OWN it? Did they place it, write it, or make it — is it content? → It's on the desk. Is it a control, indicator, or container the app provides? → It's terminal.
2. Would a state change benefit from being physically obvious? The skeuomorphism's whole job is making state changes legible in a playful way — bound/torn/pinned, lifted/settled, locked-pressed. If the element has meaningful states the user must track at a glance, the desk's hardware vocabulary is available. Terminal state is shown by color/fill only.
3. Would the metaphor ever lie? If the physical story misrepresents what the system does, the metaphor is FORBIDDEN even on the desk — this is why delete lifts away (the trash keeps things whole) and only Empty Trash burns (destruction is real there). An honest terminal beats a lying desk, always.

3 · The catalog — every example in the app

THE DESK (material, beats allowed)
· images, ink, drawn shapes, canvas text — the art itself
· paper notes: ring-bound page · taped sticky · push-pinned landmark; torn edge persists (Note Lifecycle)
· object icons: star·pin·flag·heart·bolt·leaf, top-light gloss (Icon Doc t3–t6)
· the red glossy pin — exactly three homes, all on paper
· tape, torn edges, binder rings — state hardware
· beats: lift·settle·nudge·press·tear·glide + dive, bloom, making-room, burn-on-purge (Interaction Physics)
· the lit stage growing as the world grows
· skeuo map shapes (future decorations epic, sketched)
· seed/demo art (commissioned, never generated)
THE TERMINAL (flat mono, opacity only)
· rail, dock, charm bar, path bar — unicode glyphs, flat chips
· menus, tooltips (five shapes), toasts, dialogs
· search track, facet pills, tag chips — mono type
· takeovers: graph, outline, gallery, trash, settings chrome
· selection outline, smart guides, marquee, lens ring — screen-constant 2px
· hint charms (page/frame corners) — drawn bordered divs, flat
· the flat teardrop pin in path/menus — "place," spoken in terminal
· progress (import strip), counts, shortcuts — tabular mono
· frame borders + title/sort chip (furniture: exists or doesn't)

4 · The decided border cases — don't re-argue these

· The appearance switcher shows object icons at chrome size — they remain world assets on loan; the popover itself is terminal.
· The source panel shows another world's content — panel chrome is terminal; the thumbnails inside are desk.
· Notes are paper even though panels are chrome: the page is world content the user wrote; its container behaviors (clamp, z-rung) are terminal duties performed invisibly.
· The shadow follows the glass, nothing else — a shadow marks viewport-floating (sticky, drag ghost, menus); world content at rest is FLAT (→ RFC §8.5).
· The frame is terminal furniture even though it holds desk objects — it's a container the app provides; its making-room slide is the members moving (desk), not the frame emoting.
· Hint charms stay flat even on pictures — they are annotations ON world content, not part of it.

5 · Forbidden moves — the PR checklist

□ no gradient, gloss, bevel, or shadow-at-rest in chrome — terminal is flat
□ no drawn/icon-font glyph where the surface's unicode set exists — and no emoji anywhere
□ no beat on chrome: chrome animates OPACITY only, on the one shared clock
□ no looping/ambient animation anywhere — one beat per user act (iteration-count ≠ 1 fails review)
□ no scale outside ±1% in beats; no bounce, no rubber band, no spring
□ no metaphor that lies: nothing crumples/burns/shatters unless the data is truly destroyed
□ no raw hex — every color is an --ew-* token first; no literal z-index — rungs only
□ no new fonts (Maple Mono in note text is THE exception, already spent)
□ no per-element size thresholds — EW_FURNITURE_MIN_PX / EW_PAGE_FLOOR_PX only
□ no shadow on world content at rest; --ew-drag-shadow rides lift→settle only
□ never the word "file" in UI copy; impact stated as fact before multi-place actions

6 · When you think the rule is wrong

Sometimes it will be. Route it: a turn in the relevant design document (or a new one) → an entry in the design-team letter → DESIGN-QUEUE → RFC rev bump → this rulebook amends. What you may not do is ship the exception and let the document drift — the rulebook is only worth its enforcement.
